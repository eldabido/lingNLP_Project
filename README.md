# Детекция нецензурных никнеймов: Rule-based vs ML

Сравнительное исследование подходов к обнаружению нецензурной лексики в никнеймах на образовательной платформе.

Сравниваем:
- **Go rule-based система** — нормализация + Ахо-Корасик + стемминг
- **Собственные ML-модели** (Python) — TF-IDF на символьных n-граммах + классификаторы
- **Готовые NLP-модели** — rubert-tiny2-sentiment, rubert-tiny-toxicity, detoxify

## Наши ML-модели (model.py)

Три классификатора, обученных на нашем датасете. Все используют одинаковый pipeline обработки.

### Pipeline обработки

```
Исходный текст → Нормализация → TF-IDF (char n-grams) → Классификатор → 0/1
```

**Шаг 1: Нормализация.** Перед подачей в модель каждый текст проходит через Python-порт нормализатора из Go (normalizer.py). Порядок этапов идентичен Go-реализации: замена цифр → гомоглифы → транслитерация → только буквы → схлопывание повторов. Это позволяет сравнивать ML-модели с Go-системой на одинаково обработанных данных.

**Шаг 2: TF-IDF с символьными n-граммами.** Ключевой выбор в проекте — использование `TfidfVectorizer` с параметром `analyzer="char_wb"` и `ngram_range=(2, 5)`. Это означает, что каждый текст разбивается не на слова, а на последовательности из 2–5 символов с учётом границ слов.

Почему символьные n-граммы, а не словесные:
- Никнеймы — это не предложения, а короткие строки, часто из одного слова; словесные признаки (bag-of-words) не работают, если слово не встречалось в обучении.
- Символьные n-граммы улавливают характерные *подстроки* нецензурных корней.
- После нормализации обфусцированные варианты приводятся к тому же набору n-грамм, что и исходные слова.

TF-IDF (Term Frequency — Inverse Document Frequency) взвешивает каждый n-грамм: частые n-граммы (встречаются во всех текстах) получают низкий вес, редкие и характерные — высокий. Параметр `sublinear_tf=True` применяет логарифмирование (`1 + log(tf)` вместо `tf`), что сглаживает влияние очень частых n-грамм.

**Шаг 3: Классификатор.** На выходе TF-IDF — разреженный вектор из ~5000–10000 признаков (каждый — вес одного n-грамма). Этот вектор подаётся на вход классификатору.

### Logistic Regression (logreg)

Линейная модель, которая для каждого n-грамма учит вес: положительный вес означает, что n-грамм характерен для нецензурных слов, отрицательный — для нейтральных. Финальное решение — взвешенная сумма всех n-грамм через сигмоиду.

Параметры:
- `C=1.0` — сила регуляризации (L2 по умолчанию), предотвращает переобучение.
- `class_weight="balanced"` — автоматическая балансировка классов, чтобы модель не смещалась в сторону более частого класса (запрещённых слов в датасете больше).
- `max_iter=1000` — максимум итераций оптимизатора.

### LinearSVC (svm)

Метод опорных векторов (SVM) с линейным ядром. Ищет гиперплоскость, которая максимально разделяет два класса в пространстве TF-IDF-признаков. В отличие от LogReg, оптимизирует не логистическую функцию потерь, а hinge loss, что делает модель более устойчивой к выбросам.

Обёрнут в `CalibratedClassifierCV` — это позволяет получать вероятности (predict_proba), которых у голого LinearSVC нет. Калибровка обучается методом Platt scaling на held-out данных внутри кросс-валидации.

Параметры: `C=1.0`, `class_weight="balanced"`.

### RandomForest (rf)

Ансамбль из 200 деревьев решений, каждое обучается на случайном подмножестве данных и признаков (bagging). Финальное предсказание — голосование большинством.

Параметры:
- `n_estimators=200` — количество деревьев в ансамбле.
- `max_depth=None` — деревья растут до полного разделения (без ограничения глубины).
- `class_weight="balanced"` — балансировка классов.

RandomForest менее подходит для текстовых данных с высокой размерностью (тысячи n-грамм-признаков), поэтому ожидаемо уступает линейным моделям.

### Оценка: стратифицированная кросс-валидация

Для оценки качества используется **5-fold стратифицированная кросс-валидация** (`StratifiedKFold`). Датасет делится на 5 частей с сохранением пропорции классов; модель 5 раз обучается на 4 частях и тестируется на оставшейся. Итоговые метрики — среднее ± стандартное отклонение по 5 фолдам.

Тест-кейсы из Go (66 штук) **исключены из обучающей выборки** и используются только для финальной оценки — это предотвращает утечку данных (data leakage).

## Готовые NLP-модели (baselines.py)

Три предобученные модели, используемые «как есть» (без дообучения на нашем датасете).

### rubert-tiny2-russian-sentiment

**Модель:** `seara/rubert-tiny2-russian-sentiment` (HuggingFace)
**Архитектура:** RuBERT-tiny2 (~29M параметров) — компактная версия BERT, предобученная на русских текстах, затем дообученная (fine-tuned) на задачу классификации тональности.
**Данные обучения:** объединение датасетов Kaggle Russian News, Linis Crowd, RuReviews, RuSentiment.
**Классы:** neutral, positive, negative.
**Как используется:** текст классифицируется моделью; если предсказанный класс = `negative`, считаем никнейм запрещённым.

**Ограничения для нашей задачи:**
- Sentiment ≠ toxicity: фраза «плохой фильм» — negative по тональности, но не оскорбление.
- Модель обучена на полноценных предложениях, а не на коротких никнеймах — контекста для классификации может не хватать.
- Не обрабатывает обфускацию — BERT-токенизатор разбивает такие строки на подслова, теряя смысл.

### rubert-tiny-toxicity

**Модель:** `cointegrated/rubert-tiny-toxicity` (HuggingFace)
**Архитектура:** RuBERT-tiny (~12M параметров), дообученный на классификацию токсичности.
**Данные обучения:** комментарии с 2ch, pikabu, ok.ru — реальные русскоязычные данные с обсценной лексикой.
**Классы:** non-toxic, insult, obscenity, threat, dangerous.
**Как используется:** модель выдаёт вероятности по каждому классу; если вероятность `non-toxic` < 0.5, считаем никнейм запрещённым.

**Почему эта модель наиболее релевантна:** она обучена именно на определение оскорблений (insult) и обсценной лексики (obscenity) в русском тексте — это напрямую совпадает с нашей задачей.

**Ограничения:** обучена на комментариях (длинные тексты), а не на никнеймах (1–3 слова); не обрабатывает обфускацию.

### detoxify

**Модель:** `detoxify` (Unitary AI), вариант `multilingual`
**Архитектура:** XLM-RoBERTa — мультиязычная версия RoBERTa (~278M параметров), предобученная на текстах ~100 языков.
**Данные обучения:** Jigsaw Toxic Comment Classification Challenge (Kaggle) — комментарии из Википедии, размеченные на toxicity, severe_toxicity, obscene, threat, insult, identity_attack.
**Как используется:** модель выдаёт score toxicity от 0 до 1; если ≥ 0.5, считаем никнейм запрещённым.

**Ограничения:** самая тяжёлая модель (~250 МБ); заточена под длинные комментарии; обучена преимущественно на английских данных с мультиязычным дообучением.

### substring baseline

Простейшая система без ML: проверяет, содержится ли хоть одно слово из blacklist (config.json) в lowercase-версии текста. Без нормализации. Служит нижней границей — показывает, какой результат даёт наивный подход.

### Какие решения ещё рассматривались

- **dostoevsky** — FastText для русского sentiment analysis. Не устанавливается на Windows (зависимость fasttext==0.9.2 требует C++ компилятор). Заменена на rubert-tiny2-sentiment.
- **better-profanity** — работает только с английским словарём; сравнивать на русских данных некорректно.
- **profanity-filter** — поддерживает русский через pymorphy2, но нестабильно устанавливается, давно не обновляется.
- **Perspective API** (Google Jigsaw) — онлайн-API, требует ключ и интернет.
- **s-nlp/russian_toxicity_classifier** — полный RuBERT, тяжелее rubert-tiny, но потенциально точнее.

## Метрики

**F1-macro** — основная метрика. Вычисляет F1 отдельно для каждого класса (запрещённый и разрешённый), затем берёт среднее. Это важно при дисбалансе классов: если бы мы использовали accuracy, модель, которая всё запрещает, получила бы высокий балл просто потому, что запрещённых примеров больше.

**Precision** (точность) — доля верных среди предсказанных как запрещённые. Высокий precision = мало ложных срабатываний.

**Recall** (полнота) — доля найденных среди всех реально запрещённых. Высокий recall = мало пропусков.

Метрики считаются macro-averaged и в Go, и в Python — результаты сопоставимы напрямую.

## Результаты и выводы

### Go rule-based
- Отлично справляется с обфускацией (нормализация + Ахо-Корасик)
- 100% accuracy на собственных тест-кейсах
- Проседает на словах вне blacklist и незамапленных цифрах

### Наши ML-модели (LogReg, SVM)
- Обобщают на новые слова, которых не было в словаре
- Кросс-валидация: SVM F1 ≈ 0.93, LogReg F1 ≈ 0.90
- Дают false positives на словах-ловушках

### Готовые BERT-модели
- rubert-tiny-toxicity наиболее релевантна — обучена именно на русскую токсичность
- Все BERT-модели не обрабатывают обфускацию без нормализации
- Обучены на комментариях, а не на коротких никнеймах — domain mismatch

### Общий вывод
Ни один подход не решает задачу идеально. Rule-based отлично работает с обфускацией, но ограничен словарём. ML обобщает, но даёт ложные срабатывания. BERT понимает семантику, но не заточен под никнеймы. Оптимальный подход — комбинация: нормализация + rule-based для известных слов, ML для обобщения на новые.
